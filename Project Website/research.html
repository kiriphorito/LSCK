Research


In this page we will talk about the technologies that we are using and why?

Visual Studio Extension

One of the very first things we had to research was how to develop a Visual Studio extension and already we as a team need to make a decision. Should the extension be developed in C# or Visual Basic. With C#, neither of us has used it before, but with VB one of the members have.

After research around extensions and the two languages, we came to the conclusion that C# would be the better option for the following reasons:

  • Many of the open source code which we can use are in C#
  • It interfaces with more APIs in comparisons to Visual Basics
  • Some operations in C# are faster
  • C# is similar to other C languages and we studied C last year

There are also a larger number of packages available to use for C# than VB. Some that we are looking into are Json.NET and SSH.NET.

Data Model

For the back-end of the extension, we needed a way to store data externally easily and be able to read it back into the extension. At first we considered just using a plain txt file with delimiters but it was apparently that this was both inefficient and inconvenient. So we decided to use a pre-existing data structure or easy way to make a data strucutre. The two we considered are XML or Extensible Markup Language or JSON JavaScript Object Notation.

Both of them are similar in terms of:

  • Being readable in plain text form
  • Being hierarchical
  • Having APIs for a variety of different languages and so can be used across programming platforms

But there are reasons why JSON is better than XML. There are as follows:

  • JSON is quicker read and write to
  • It is also shorter
  • Supports arrays
  • Easier to parse than XML

Also a member of the team knows and understands the JSON data structure. Because of this and the points above, we decided to use JSON for the data structure.

Front-end Framework

Moving onto the website portfolio, it was decided that we would use HTML and CSS for the base of the website. What is next to be considered is potentially which front-end framework should be used. There are a number of them available to use but the three we considered were:

  • Bootstrap
  • Foundation
  • Skeleton

First up is Skeleton. The framework is very simple and lightweight and you'd have to spend a lot of time in developement if you want to create a good looking website. But the website we hoped to generate is simple so Skeleton became a potential framework to use.

Foundation and Bootstrap on the other hand are loaded with examples and loads of features. Both are considered to be very good and popular among developers. Because of this there is also a lot of support and help online for both frameworks unlike Skeleton. But is it said that Bootstrap is better when it comes to cross platform developement. Also both members of the team have used Bootstrap before and have experience in it.

As a result of there being more support, features and that members have used this framework before, we selected to used Bootstrap.

Hosting Operating System

Next up is the choice of which operating system to support. This project being a Microsoft project, we had to consider Microsoft hosting which accounts for almost a third of the hosting on the internet. But there is also the LAMP stack or Linux hosting which accounts for almost two thirds of the market share.

There are not really any technologies that are cross platform between the two and that goes for web hosting. Apach and NGINX are all Linux based.

Unfortunately, one team member is experienced in setting up a LAMP stack. And no-one has used Microsoft hosting before. So due to lack of experience, lower market share, and generally more expensive than Linux (as Linux is open source), we decided to use Linux. It wasn’t a requirement to use Microsoft hosting.

Although support for Microsoft hosting could be a potential future implementation.

Hosting Software

With the fact that we are using Linux, we now need to decide which web hosting software we want to use. There are two major competitors in this market. Apache and NGINX. Both are very popular on the internet with Apache losing market share to NGINX. Both have their pros and cons and are neck and neck with each other.

Apache is older and more developed with a lot of support and experienced users. It also supports dynamic content natively like PHP and doesn’t need to be processed externally. And has a lot of support of interpreted languages. But cannot deal with many concurrent requests very well as it slows down under load.

NGINX is the exact opposite. Younger software after being released in 2010 rather than the 1990s. Has to outsource dynamic content to another piece of software. But can deal with a lot of concurrent requests meaning performance can be estimated and it works as should with many users. It is also very lightweight and less resource intense compared to Apache.

Again like with all the other decision, one member of the team has experience with Apache, and with the potential to use PHP in the future Apache was the software of choice.

GitHub

We made use of GitHub as our git repository to commit our changes to the project and create our repository. This is due to the fact that the team had worked together before using the platform, and had no previous issues with it. There are others like GitLab and BitBucket, but both members of the team are confortable with GitHub.

Atom

We used atom to write the html and css for the website. Both members of the team used this text-editing software as their standard coding tool, so we picked it over other choices like sublime text.

Photoshop

We chose to use photoshop to create mockups of what the extension would eventually look like. Photoshop was picked over other image editing software, as it is very intuitive and we had prior experience with it.

draw.io

To develop our flow diagram we used draw.io, a very useful software for designing any kind of diagrams, specially UML and flow charts.

Slack

Slack was used as a means of communication for the team. All important news and changes to the project were notified through here, and it has proven very useful up until now.
