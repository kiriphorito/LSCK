Achievements and Implementations


In this page we will discuss about achievements and implementations that haven't been mentioned in other parts of the project. For instance design patterns are in Architecture.

Concurrency and Threading

We had issues updating the UI when a new solution was opened, and when a solution was closed. On top of this, Visual Studio would crash and throw errors, so we decided to implement threading to solve the problema. This is used to constantly check for changes in the open solution and the generated website. Threads are created from the constructors of the tool windows and check every certain time for a particular change in a static class, which is used as a bridge connecting the different tool windows. This bridge class enables communication between the Visual Studio IDE and the actual UI, updating the latter as needed when changes are detected. Thanks to this, the extensión knows at all times when a new solution has been opened, or when the existing one is closed.

This however derived in another issue, which was the fact that the extensión would throw many errors if no solution was open. For this purpose, and knowing that LSCK is of no use when no solution is open, we decided to combine the detection of the closing solution to an event in the UI that would blank out the entire extensión, not allowing the user to perform any actions.

When uploading and previewing the website, Visual Studio hung and this also gave us some errors. Threading was also used to solve this problema, as a thread running in the background would check every certain amount of time if the website had been updated (again, through the bridge class).

Extractor Class

For dealing with the automatic extraction of code, we initially couldnt find a way of implementing something relatively simple. To solve this problema we built the extractor class, which runs through the entire solution opened by the user, looking for specific character sequences. It will add all snippets (commented in the way the LSCK extension requires) to the website the app automatically generates. This extractor class only supports the languages specified in the code, which in the case of the LSCK extension, are the main languages supported by Visual Studio.

UI Design

We believe we have designed a UI that fits well with the Visual Studio default design. When developing the XAML, we didn’t want to make it look too fancy, as it would not sink in too well with the rest of the Visual Studio IDE. The fact that the extension is divided into tabs which have clear purposes for working towards the automatically generated website is good for usability, and helps the user understand the way the extension works and familiarize quickly. We also divided the UI into three key tool windows so the funtions were not overloaded into one single window. Hence, each tool window has a distinct and clear purpose, whereby the LSCK window is responsible for the overall addition of code and website customization, the Structure window enables the user to reorganize previously added snippets and sections, and modify comments, and the Preview tool window acts as a browser to help the user get a feeling of what the site will look like before actually uploading.
